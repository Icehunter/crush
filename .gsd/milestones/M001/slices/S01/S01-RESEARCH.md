# S01: DB Schema + SQLC Queries — Research

**Date:** 2026-03-27
**Depth:** Targeted (known technology, new tables in established codebase)

## Summary

This slice adds three new SQLite tables (`milestones`, `slices`, `tasks`) to Crush's existing database, a goose migration file, SQLC query files for CRUD operations, and regenerated Go code. The codebase has a well-established pattern for all of this: goose migrations in `internal/db/migrations/`, SQLC queries in `internal/db/sql/`, generated code in `internal/db/`, and an `embed.FS` that auto-discovers migration files. No new technology is involved — this is applying the exact same patterns used for sessions, messages, files, and read_files.

The primary risk is getting the schema design right, since M002-M004 all build on it. The key design decisions: TEXT primary keys (matching sessions pattern), TEXT status/phase enums (matching `TodoStatus` pattern), integer ordering columns, and foreign keys between the three tables. The `sessions.parent_session_id` column already exists and will be the link point for M002 (sessions → milestones).

## Recommendation

Follow the existing patterns exactly. One migration file, one SQLC query file per table (or one combined `auto.sql`), then `sqlc generate`. Store status and phase as TEXT columns — this matches the `TodoStatus` pattern in `session.go` and avoids CGO integer-enum complications. Use TEXT UUIDs for primary keys (matching sessions). Add `sort_order INTEGER` to slices and tasks for deterministic ordering. Include `created_at`/`updated_at` INTEGER columns with triggers (matching existing tables).

## Implementation Landscape

### Key Files

- `internal/db/migrations/20260327000000_add_auto_tables.sql` — New goose migration creating `milestones`, `slices`, `tasks` tables with indexes, triggers, and foreign keys. Follows pattern from `20260127000000_add_read_files_table.sql` and `20250424200609_initial.sql`.
- `internal/db/sql/milestones.sql` — SQLC queries for milestones: Create, Get, List, Update status/phase, Delete. Follows `sessions.sql` pattern.
- `internal/db/sql/slices.sql` — SQLC queries for slices: Create, Get by ID, List by milestone, Update status/phase, Delete. Follows `read_files.sql` pattern for filtered listing.
- `internal/db/sql/tasks.sql` — SQLC queries for tasks: Create, Get by ID, List by slice, Update status/phase, Delete, get active/next task.
- `internal/db/embed.go` — No change needed. `//go:embed migrations/*.sql` already picks up any new `.sql` file.
- `internal/db/connect.go` — No change needed. `goose.Up()` auto-runs new migrations.
- `sqlc.yaml` — No change needed. Already configured to read `internal/db/sql/*.sql` and output to `internal/db/`.
- `internal/db/*.go` (generated) — Will be regenerated by `sqlc generate`. New files: `milestones.sql.go`, `slices.sql.go`, `tasks.sql.go`. Updated files: `models.go` (new structs), `db.go` (new prepared statements), `querier.go` (new interface methods).

### Schema Design

```sql
-- milestones
CREATE TABLE milestones (
    id TEXT PRIMARY KEY,
    title TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'pending',  -- pending, active, completed, failed
    phase TEXT NOT NULL DEFAULT 'pre_planning',  -- pre_planning, planning, executing, summarizing, validating, completed
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL
);

-- slices
CREATE TABLE slices (
    id TEXT PRIMARY KEY,
    milestone_id TEXT NOT NULL,
    title TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'pending',
    phase TEXT NOT NULL DEFAULT 'pre_planning',
    sort_order INTEGER NOT NULL DEFAULT 0,
    depends_on TEXT,  -- comma-separated slice IDs or JSON array
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL,
    FOREIGN KEY (milestone_id) REFERENCES milestones (id) ON DELETE CASCADE
);

-- tasks
CREATE TABLE tasks (
    id TEXT PRIMARY KEY,
    slice_id TEXT NOT NULL,
    milestone_id TEXT NOT NULL,
    title TEXT NOT NULL,
    status TEXT NOT NULL DEFAULT 'pending',
    phase TEXT NOT NULL DEFAULT 'pre_planning',
    sort_order INTEGER NOT NULL DEFAULT 0,
    description TEXT,
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL,
    FOREIGN KEY (slice_id) REFERENCES slices (id) ON DELETE CASCADE,
    FOREIGN KEY (milestone_id) REFERENCES milestones (id) ON DELETE CASCADE
);
```

### Build Order

1. **Migration file first** — This is the foundation. Write the DDL, verify it applies cleanly to a fresh DB and to an existing DB. The entire slice depends on this being correct.
2. **SQLC query files** — Write the SQL queries for all three tables. Run `sqlc generate` to produce Go code. This proves the schema and queries are syntactically valid together.
3. **Verify compilation** — `go build ./internal/db/...` must pass. The generated code must compile.
4. **Write a smoke test** — A test in `internal/db/` (or `internal/auto/`) that opens an in-memory SQLite DB, runs migrations, and exercises the generated CRUD queries. This proves the schema works end-to-end.

### Verification Approach

1. `sqlc generate` — must succeed with no errors (proves schema + queries are valid).
2. `go build ./internal/db/...` — generated code compiles.
3. `go test ./internal/db/...` — if a test is added, it passes.
4. Manual inspection: `sqlite3 :memory:` with the migration SQL to verify table creation, foreign keys, indexes.

## Constraints

- `CGO_ENABLED=0` — SQLite via `modernc.org/sqlite` (already in deps). No C extensions available.
- Migration naming: `YYYYMMDD000000_description.sql` — latest is `20260127000000`. New one should be `20260327000000` or later.
- SQLC v1.30.0 — this is the version generating existing code. Must match.
- `emit_prepared_queries: true` in `sqlc.yaml` — every new query adds a prepared statement field to `Queries` struct. This is automatic but adds to `Prepare()`, `Close()`, and `WithTx()` boilerplate (all generated).
- All timestamps are Unix seconds (not milliseconds — the initial migration comment says milliseconds but `strftime('%s', 'now')` returns seconds). Follow the existing pattern: store as `INTEGER`, use `strftime('%s', 'now')` in triggers.

## Common Pitfalls

- **Timestamp confusion** — The initial migration comments say "Unix timestamp in milliseconds" but `strftime('%s', 'now')` returns seconds. The codebase actually uses seconds. Follow the actual behavior (seconds), not the comments.
- **Foreign key enforcement** — Pragmas in `connect.go` set `foreign_keys = ON`. This means INSERT order matters in tests: milestones before slices before tasks. Deletion cascades from parent to child.
- **SQLC query naming** — Names must be unique across ALL `.sql` files (not just per-file). E.g., can't have `Create` in both `milestones.sql` and `slices.sql` — must be `CreateMilestone`, `CreateSlice`.
- **`RETURNING *`** — Use for `:one` queries that need the created/updated row back. This is the established pattern (see `CreateSession`).
- **`sql.NullString` for nullable TEXT columns** — SQLC generates `sql.NullString` for nullable columns, `string` for NOT NULL. Keep columns NOT NULL where possible to avoid null-handling complexity in Go.
